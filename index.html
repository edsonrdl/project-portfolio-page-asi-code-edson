Kafka e CDC: Integração de Dados em Tempo Real

Com a crescente demanda por dados em tempo real, arquiteturas orientadas a eventos têm se tornado essenciais. O Apache Kafka, uma plataforma distribuída, é capaz de lidar com grandes volumes de dados, fornecendo escalabilidade e alta disponibilidade.

Change Data Capture (CDC) monitora alterações em bancos de dados e, com Kafka Connect e Debezium, é possível capturar essas mudanças de sistemas legados e transmiti-las em tempo real para um destino, como o PostgreSQL. Neste exemplo, foi configurado um ambiente usando Docker Compose, onde o MySQL atua como fonte de dados, o PostgreSQL como destino e o Redis como exemplo adicional de uso.

O Kafka Connect desempenha um papel fundamental na integração de microserviços ao facilitar a troca de dados entre sistemas novos e legados. Isso suporta padrões como DDD (Domain-Driven Design) e CQRS (Command Query Responsibility Segregation), permitindo a separação de comandos e consultas em diferentes bancos de dados.

Considere uma aplicação que precisa sincronizar dados entre o MySQL e o PostgreSQL. O Kafka Connect resolve esse desafio ao capturar alterações no MySQL e enviá-las ao PostgreSQL em tempo real, evitando inconsistências e garantindo a integridade dos dados.

Escrito em Java e Scala, o Kafka é altamente configurável. Durante os testes, Docker Compose foi utilizado para criar e gerenciar containers, simplificando o processo de configuração e tornando o ambiente facilmente replicável.

Benefícios do CDC com Kafka Connect

  • Automação: Alterações no MySQL são automaticamente refletidas no PostgreSQL.
  • Escalabilidade: O Kafka facilita a adição de novos destinos (sinks), como Redis para cache.
  • Modularidade: Em arquiteturas de microserviços, o Kafka Connect permite uma integração desacoplada.

O Kafka e o Kafka Connect são ferramentas robustas para sincronização de dados em arquiteturas orientadas a eventos, suportando padrões como CQRS e DDD. O uso de Docker Compose simplifica a criação e o teste de ambientes de CDC.

Projeto: Sistema de Processamento de Pedidos com RabbitMQ

Este projeto implementa um sistema de processamento de pedidos usando uma arquitetura baseada em microserviços com RabbitMQ como sistema de mensageria. A comunicação entre os serviços é realizada através de mensagens roteadas pelo *Direct Exchange*, permitindo a distribuição de tarefas específicas de acordo com as *routing keys*.

Arquitetura Geral

O sistema consiste em uma API ASP.NET Core que atua como publicadora de mensagens, enquanto os microserviços em Node.js (OrderProcessorService, PaymentService e NotificationService) atuam como consumidores. Cada mensagem é roteada para uma fila específica com base nas *routing keys* definidas no *Direct Exchange* do RabbitMQ. As filas principais são:

  • order_queue: Processa novos pedidos recebendo mensagens com a *routing key* order.new.
  • payment_queue: Processa pagamentos recebendo mensagens com a *routing key* order.payment.
  • notification_queue: Envia notificações para usuários com mensagens com a *routing key* order.notification.

Microserviços e Bancos de Dados

Cada microserviço possui seu próprio banco de dados, permitindo isolamento de dados e maior flexibilidade. Isso ajuda a garantir que cada microserviço seja autônomo e desacoplado dos demais, facilitando a escalabilidade e a manutenção do sistema. Os bancos de dados utilizados são:

  • OrderProcessorService: Armazena informações relacionadas aos pedidos processados.
  • PaymentService: Gerencia informações de transações de pagamento.
  • NotificationService: Armazena e gerencia dados relacionados a notificações enviadas aos clientes.

Benefícios da Arquitetura

Os principais benefícios dessa arquitetura incluem:

  • Escalabilidade: Cada microserviço pode ser escalado individualmente conforme a demanda.
  • Modularidade: Os serviços são desacoplados, facilitando modificações e adições sem impactar outros serviços.
  • Roteamento Determinístico: As mensagens são roteadas para as filas apropriadas com base em *routing keys*, proporcionando controle granular sobre a distribuição das mensagens.
  • Isolamento de Dados: Cada microserviço possui seu próprio banco de dados, garantindo segurança e integridade dos dados.
